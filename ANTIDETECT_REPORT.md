*Апрель 2026*

## 1. Как работает детекция на платформах

### YouTube Content ID

YouTube использует **два независимых отпечатка** — аудио и видео. Они обрабатываются отдельно: клип может быть задетектирован только по аудиодорожке, даже если видео сильно изменено.

**Аудиоотпечаток** строится на частотном анализе (спектральные признаки). Система устойчива к:
- Изменению скорости ±5% → матч всё равно находится
- Шуму (до 45% громкости фонового шума)
- Ресэмплингу
- MP3 конвертации
- Лёгкой коррекции тона

Только при изменении скорости **свыше 5%** (pitch-corrected) матч начинает ломаться стабильно. Простой ресэмплинг без pitch correction не помогает — система использует pitch-invariant признаки.

**Визуальный отпечаток** (perceptual hash) устойчив к:
- Кропу и ресайзу
- Изменению насыщенности/яркости
- Добавлению логотипов/текста
- Горизонтальному флипу (в некоторых реализациях)

### TikTok

TikTok использует многоуровневую систему (2024–2025):

1. **Perceptual hashing** — сравнение визуального содержимого попиксельно после нормализации
2. **Deep learning модели** — анализируют нарративный поток: как нарезаны сцены, переходы, ритм монтажа
3. **C2PA metadata tracking** — встроенные метаданные о происхождении файла, устройстве записи, истории редактирования
4. **Аудиофингерпринт** — аналогичен YouTube, матчит треки даже при лёгких изменениях
5. **Поведенческие паттерны** — анализ аккаунта: частота постинга, источники трафика, история

## 2. Оценка текущего AntiDetect

### Что сейчас делает `antidetect.py`

| Фильтр | Параметры | Оценка эффективности |
|---|---|---|
| Zoom crop | 2–4% (varies by hash) | ⚠️ Слабо — perceptual hash устойчив к кропу |
| Speed variation | ±1.5% через `setpts` | ❌ Недостаточно — порог детекции ~5% |
| Saturation | ±8% | ⚠️ Слабо — color shift не ломает хеш |
| Audio resample | `asetrate=44100*speed` | ❌ Почти не помогает — pitch-invariant системы |

**Итого**: текущий AntiDetect обеспечивает **минимальную защиту** от YouTube Content ID и **очень слабую** от TikTok. Параметры слишком консервативны для реального обхода детекции.

## 3. Что реально работает (и не ломает качество)

### 3.1 Скорость — главный рычаг

Изменение скорости с **pitch correction** — самая эффективная одиночная техника:

```
# Текущий подход (неэффективен для аудио):
setpts=1/speed*PTS  +  asetrate=44100*speed,aresample=44100

# Эффективный подход — tempo change без pitch shift:
[video] setpts=1/speed*PTS
[audio] atempo=speed   ← сохраняет pitch, меняет темп
```

Изменение tempo на **±5–8%** надёжно ломает аудиофингерпринт при сохранении нормального звука. Фильтр `atempo` (диапазон 0.5–2.0) специально создан для этого.

### 3.2 Кроп — увеличить масштаб

Кроп **4–6%** с последующим scale заметнее меняет хеш. Важно делать кроп **несимметричным** (не с центра) — это разрушает паттерн сильнее.

### 3.3 Шум в видео

Добавление лёгкого цифрового шума (`noise` фильтр) помогает против perceptual hashing:

```
noise=alls=8:allf=t
```

Шум `alls=8` визуально незаметен, но меняет пиксели достаточно чтобы сдвинуть хеш.

### 3.4 Метаданные

Платформы читают `creation_time`, `encoder`, `device model` из контейнера. Нужно стрипать все метаданные и инжектировать случайные:

```
-map_metadata -1               # удалить все метаданные
-metadata creation_time="..."  # случайное время создания
```

### 3.5 Флип (осторожно)

Горизонтальный флип (`hflip`) технически ломает многие perceptual hash реализации. Минус: для сериалов с субтитрами текст станет зеркальным — неприемлемо. Для видео без текста — один из лучших инструментов.

### 3.6 Интро-фрагмент

Добавление 2–4 секунды случайного контента в начало или конец видео меняет длину и смещает все временные отпечатки. Это особенно эффективно против систем, привязанных к timestamp-based fingerprinting.

## 5. Главные уязвимости — что НЕ обходится

**TikTok Deep Learning (2025)**: анализирует нарративный поток — порядок сцен, ритм монтажа. Если вы нарезаете один и тот же эпизод одинаково каждый раз → паттерн детектируется. Обход: менять точки нарезки от запуска к запуску (уже частично решено через overlap + silence-based chunking).

**YouTube Audio при ±5% с pitch correction**: работает, но `atempo` влияет на восприятие. При 106% скорости диалоги звучат чуть быстрее — приемлемо для большинства контента.

**C2PA metadata**: это метаданные, встроенные при записи оригинала. Стриппинг через `map_metadata -1` убирает их полностью. Если платформа требует C2PA-подпись для монетизации в будущем — это проблема, но не сейчас.

## 6. Рекомендации для MVP+

### Приоритет 1 — Аудио (критично для YouTube)

Заменить в `antidetect.py`:
```python
# Было:
af = f"asetrate=44100*{speed},aresample=44100"

# Станет (tempo без pitch shift):
af = f"atempo={speed}"
```

Увеличить диапазон скорости с **±1.5%** до **±5–6%**:
```python
speed_var = 0.05   # вместо 0.015
speed = 1.0 + v * speed_var  # 0.95 — 1.05
```

### Приоритет 2 — Метаданные (быстро, бесплатно)

Добавить в ffmpeg команду:
```python
"-map_metadata", "-1",
"-metadata", f"creation_time={random_datetime}",
```

### Приоритет 3 — Видео шум

Добавить в `vf`:
```python
f"noise=alls={noise_level}:allf=t"
```
где `noise_level` = 5–10, варьируется per-clip через хеш.

### Приоритет 4 — Кроп увеличить

С текущих 2–3% до 4–6%, с несимметричным сдвигом центра.

## 7. Реалистичный прогноз

| Платформа | Текущий риск | После улучшений |
|---|---|---|
| YouTube (аудио матч) | Высокий | Средний |
| YouTube (видео матч) | Средний | Низкий |
| TikTok (дублирование) | Высокий | Средний |
| TikTok (AI паттерн) | Средний | Средний (не решается техникой) |

**Главный вывод**: анти-бан техники снижают вероятность детекции, но не устраняют риск полностью. Наибольший риск — TikTok Deep Learning, который анализирует нарративную структуру контента. Технические трансформации помогают с хешами, но не с AI-анализом паттернов монтажа.

Для реального масштаба нужно также варьировать: точки нарезки (уже есть через silence-based chunking), аккаунты, время постинга и регионы.

## 8. Банинг аккаунтов — отдельный уровень защиты платформ

Помимо детекции контента, TikTok и YouTube отдельно отслеживают **поведение аккаунта**. Это второй уровень защиты, независимый от видеофингерпринтов.

### 8.1 TikTok — Trust Score система

TikTok присваивает каждому аккаунту **trust score** (оценка доверия). Он формируется с первых минут после регистрации:

- Новый аккаунт сразу помечается как **unverified**
- В первые 48–72 часа идёт пристальный мониторинг
- Немедленный постинг нескольких видео после регистрации = сигнал спама = score падает → нулевые просмотры
- Нарушения уменьшают score и влияют на **все** будущие видео аккаунта, не только заблокированные

**Что триггерит бан аккаунта:**
- Частый постинг сразу после регистрации без прогрева
- Несколько аккаунтов с одного IP + одного устройства
- Перекрёстные лайки/подписки между своими аккаунтами
- Одинаковые подписи/описания видео на разных аккаунтах
- Синхронное время постинга на нескольких аккаунтах
- Использование VPN или датацентровых прокси

**Сколько аккаунтов можно на одном устройстве**: безопасно — 2–3 аккаунта. Более 3 = высокий риск. Если один аккаунт получает бан — TikTok банит все связанные аккаунты и IP.

**Устройства**: TikTok использует device fingerprinting — смена IP не помогает, если device ID, браузерный fingerprint, и поведение совпадают.

### 8.2 YouTube — Spam + инаутентичный контент

YouTube не публикует жёстких лимитов на загрузку, но поведенческие сигналы фиксируются:

| Поведение | Риск |
|---|---|
| 1–5 видео/день на новом канале | Низкий |
| 5–10 видео/день после прогрева | Допустимо |
| 10+ видео/день | Высокий риск shadow ban |
| 100+ видео за один день | Бан (throttle + потенциальный страйк) |
| Загрузка всего контента сразу + scheduling | Безопасно |

YouTube с июля 2025 обновил политику: **«inauthentic content»** (переименованный «repetitious content») — массово произведённый шаблонный контент с минимальными вариациями = запрет монетизации канала целиком.

3 страйка за 90 дней = **терминация канала**.

Что именно триггерит страйк за спам:
- Одинаковый контент на нескольких каналах
- Массовый реаплоад контента других авторов
- Автогенерированный контент без editorial value

### 8.3 Прогрев аккаунта (Account Warm-Up)

Это критически важный шаг для любого нового аккаунта перед массовым постингом:

**TikTok — рекомендуемый warm-up (7–14 дней):**
1. Дни 1–3: только просмотры, лайки, подписки на другие аккаунты. Не постить.
2. Дни 4–7: опубликовать 1–2 видео. Смотреть реакцию.
3. Дни 8–14: постепенно увеличивать до 2–3 видео/день
4. После 14 дней: возможен режим 3–5 видео/день

**YouTube — рекомендуемый warm-up (7 дней):**
1. Верифицировать канал по телефону (повышает trust)
2. Дни 1–3: заполнить профиль, загрузить 1–2 «якорных» видео
3. Дни 4–7: загружать по 1 видео/день, смотреть другие каналы, комментировать
4. После 7 дней: можно переходить к регулярному постингу

### 8.4 Стратегия для масштаба (несколько аккаунтов)

Если цель — запустить несколько аккаунтов параллельно (что необходимо для SaaS):

**Изоляция на уровне устройств/сетей:**
- Разные телефоны или антидетект-браузеры (GeeLark, Multilogin) для каждого аккаунта
- Отдельный мобильный IP на каждый аккаунт (мобильные прокси, не датацентровые)
- Разные SIM-карты или email для регистрации

**Поведенческая изоляция:**
- Разное время постинга (±2–4 часа между аккаунтами)
- Разные описания и хэштеги
- Не взаимодействовать между своими аккаунтами
- Разные часовые пояса если возможно

**Контентная изоляция:**
- Каждый аккаунт = разные серии или разные шоу
- Не загружать один и тот же клип на разные аккаунты
- Разные точки нарезки (уже решено через silence-based chunking с вариативностью)

### 8.5 Резюме по рискам аккаунт-бана

| Фактор | TikTok риск | YouTube риск | Решение |
|---|---|---|---|
| Новый аккаунт + сразу массовый постинг | Критический | Высокий | Warm-up 7–14 дней |
| Несколько аккаунтов с одного IP | Критический | Высокий | Мобильные прокси |
| Одинаковый контент на разных аккаунтах | Критический | Критический | Разные серии / разные нарезки |
| >5 видео/день на новом аккаунте | Высокий | Средний | Не превышать на старте |
| Синхронное расписание постов | Высокий | Средний | Рандомизировать время |
| Device fingerprint один для всех | Критический | Средний | Антидетект или разные устройства |
